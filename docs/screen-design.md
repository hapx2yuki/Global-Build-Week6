# CriteriaForge 画面設計

最終更新: 2026-07-21

## 設計の中心

CriteriaForge の主役は会話画面ではなく「Product Constitution」という共有可能な判断基準文書である。利用者は、AI と長く会話した感覚ではなく、曖昧だった意図が一つの文書にまとまり、自分が承認した基準で成果物を検査できる感覚を持つべきである。

画面全体は、次の三つのメンタルモデルを一貫して使う。

1. **編集机**: 中央の文書を読み、必要な箇所だけ直接直す。
2. **判断待ちの受信箱**: AI が決めてはならない論点だけが右側に届き、人間が承認する。
3. **検査台**: 作ったものを「意図・観測・根拠・差分」の順で照合する。

会話は補助的な入力手段に留める。主要な状態、出所、未決事項、作成可能性は、会話履歴を読み返さなくても文書上で理解できるようにする。

## 共通画面枠

| 領域 | 役割 | 主な shadcn/ui 部品 |
| --- | --- | --- |
| 左側 | 7段階の作業循環、案件、端末内の根拠保管領域 | `Sidebar`、`Dropdown Menu`、`Badge` |
| 上部 | 現在地、文書と評価の切替、言語、検索 | `Breadcrumb`、`Tabs`、`Tooltip`、`Command` |
| 中央 | Product Constitution または評価結果 | `Scroll Area`、`Collapsible`、`Attachment` |
| 右側 | 最重要の未決事項と、その判断材料 | `Field`、`Radio Group`、`Textarea`、`Sheet` |
| 下部 | 作成可能性または次の主要操作 | `Progress`、`Button`、`Tooltip` |

デスクトップでは左側、中央、右側を同時に見せる。狭い画面では中央を優先し、左側を `Sidebar` の画面外パネル、右側を `Sheet` に移す。主要な文書と評価結果を縮小して三列のまま押し込まない。

## 第一版の7画面

### 1. 意図の取り込み

**利用者の目的**  
文書、動画、デザインデータ、表計算、Git、自由記述を案件へ追加し、何を正とするかを決める。

**主要な構成**

- 上部に案件名と「この端末だけに保存」の状態。
- 中央に追加済みの根拠源一覧。種類、原文言語、更新日時、共有可否を表示。
- 初期状態は `Empty` で「ファイルを追加」「例を試す」の二択。
- 追加は `Attachment`、補足項目は `Field` と `Textarea`。
- 重複、読取不能、原文言語不明は行単位の `Alert`。処理中は `Skeleton` と `Spinner`。
- 端末外へ出る操作には、送信先と対象を示す確認画面を置く。

**第一版の完了条件**

- 追加した根拠源の種類と共有範囲を一目で確認できる。
- 読み取れなかった項目が、全体失敗ではなく対象単位で分かる。
- 英訳は参照用であり、原文が正であることを常に示す。

### 2. Product Constitution の形成

**利用者の目的**  
曖昧な意図を8区分の文書として読み、重要な未決事項だけを承認する。

**主要な構成**

- 中央は8区分の連続した文書。
- 各区分に「人間が承認」「原文から抽出」「AI が提案」の出所表示。
- 選択した区分だけ詳しい説明と直接編集を開く。
- 右側は、その時点で最も影響が大きい質問を一つだけ表示。
- 下部は5条件の作成可能性。未達条件は色だけでなく名称と説明でも示す。

**実装状態**  
操作可能な試作を実装済み。

### 3. 作成前の確認

**利用者の目的**  
不可逆な版を作る前に、重要な基準、根拠、矛盾、安定性を確認する。

**主要な構成**

- `Alert Dialog` に、5条件の合否と未達理由を並べる。
- 未承認の重要規則がある場合、AI が暗黙に確定せず、その規則の承認へ戻す。
- 作成できる場合は、版番号、固定される範囲、Codex へ渡る内容を明示。
- 主操作は「不可逆版を作成」、副操作は「戻る」。破壊的な色は削除用途に限定する。

**実装状態**  
操作可能な試作を実装済み。

### 4. 根拠の検査

**利用者の目的**  
評価に使われた観測が、どの原文・画面・動画時刻・表のセル・Git 差分に基づくかを確認する。

**主要な構成**

- 左側に評価項目、中央に対象成果物、右側に根拠の詳細。
- 文書は行、動画は時刻、デザインは画面と要素、表計算はシートとセル、Git はファイルと行へ移動できる。
- `Resizable` は十分な画面幅がある場合だけ使い、狭い画面では根拠詳細を `Sheet` にする。
- 根拠がない、弱い、矛盾する、閲覧権限がない、の状態を分ける。

**第一版の完了条件**

- 評価文から二操作以内で原典へ到達できる。
- 原典を開けない場合でも、何が不足しているか分かる。
- 私的な根拠本文は共有可能な憲法に混入しない。

### 5. 正式評価

**利用者の目的**  
固定した同じ憲法で成果物を評価し、合否の理由と修正すべき差分を理解する。

**主要な構成**

- 冒頭に対象版、憲法版、総合状態、独立実行回数。
- 各評価項目は `Collapsible`。閉じた状態でも合否と重要度が分かる。
- 開くと「意図・観測・根拠・差分」の4列を同じ順序で表示。
- `Tabs` で要約、評価項目、根拠を切り替える。
- 検索は `Input Group`。状態の絞り込みは `Toggle Group`。

**実装状態**  
FounderBrief の架空データによる操作可能な試作を実装済み。

### 6. Codex への修正依頼

**利用者の目的**  
失敗した評価項目だけを修正対象にし、権限と完了条件を明示して Codex へ渡す。

**主要な構成**

- 対象となる差分、変更を許す場所、変更してはならない場所、受入条件を表示。
- 利用者が対象を選ぶ場合は `Checkbox`、実行範囲は `Field` と `Select`。
- 外部変更の直前に、読む範囲、書く範囲、実行するコマンドを確認。
- 実行中は段階別の `Progress`。失敗時は入力と判断を保持し、再試行可能にする。
- 完了後は変更一覧と再評価への主操作を表示。

**第一版の完了条件**

- Codex が憲法を書き換えず、承認された成果物の範囲だけを変更する。
- 依頼文だけでなく、機械判定可能な受入条件を同時に渡す。

### 7. 再評価と前後比較

**利用者の目的**  
同じ憲法版で修正前後を比較し、意図に近づいたか、副作用がないかを確認する。

**主要な構成**

- 修正前と修正後の切替を同じ位置に置く。
- 項目ごとの状態変化を、悪化・不変・改善で示す。
- 「合格した」だけでなく、根拠が変化した箇所を表示。
- 憲法を変更した場合は同一条件比較ではないことを警告する。

**実装状態**  
FounderBrief の修正前25%から修正後100%への比較操作を実装済み。

## 状態設計

| 状態 | 表示 | 利用者が次にできること |
| --- | --- | --- |
| 初期 | 目的と最初の一手を示す `Empty` | 追加、例を試す |
| 読取中 | 対象と段階を示す `Skeleton` / `Spinner` | 中止、別作業を続ける |
| AI 提案 | 橙の出所表示と点状の背景 | 承認、直接編集、保留 |
| 人間承認済み | 緑の印と承認元 | 履歴確認、再編集 |
| 根拠不足 | 対象項目だけを停止 | 根拠追加、適用除外 |
| 矛盾 | 矛盾する原典を並べる | 正とする原典を選ぶ |
| 作成可能 | 5条件を100%で表示 | 不可逆版を作る |
| 評価不合格 | 差分と根拠を表示 | Codex 依頼、手動修正 |
| 完了 | 次の確認地点と成果物を表示 | 共有、再評価 |

通知は補助であり、重要な状態を `Sonner` だけに置かない。承認や作成の結果は、文書の出所表示や作成可能性にも恒久的に反映する。

## 視覚方針

- 基調は温かい生成り色の紙面と墨色。重要な承認操作だけに熾火色を使う。
- 根拠への参照は青、人間の承認は緑。色だけでなく形、文言、アイコンも併用する。
- 文書の見出しは明朝系、操作部品は読みやすい角ゴシック系、版や出所は等幅書体。
- 影を多用せず、罫線、余白、紙面の階層で構造を示す。
- 動きは開閉、状態変更、対象移動の理解を助ける範囲に限定する。

## 操作性と読み上げの受入条件

- キーボードだけで主要経路を完了できる。
- アイコンだけのボタンには読み上げ名と `Tooltip` がある。
- フォーカス位置が常に見える。
- モーダルと側面パネルは `Escape` で閉じ、閉じた後は起点へフォーカスが戻る。
- デスクトップ1440×900とスマートフォン390×844で横方向の意図しないスクロールがない。
- スマートフォンでは主要本文が少なくとも画面の70%を占める。
- 成否、未承認、根拠不足を色だけで伝えない。
- 破壊的または不可逆な操作は、対象と結果を説明する確認画面を経由する。
