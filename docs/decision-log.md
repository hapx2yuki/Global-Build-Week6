# 意思決定記録

## D-001: 非公開リポジトリから開始する

- 日付: 2026-07-20
- 状態: 採用
- 判断: 開発中は非公開とし、審査用アカウントへ閲覧権限を付与する方針で始める。
- 理由: 公開ライセンスの選択は権利関係に影響するため、製品と外部素材が未定の段階では決めない。
- 再確認条件: 提出前に公開へ切り替える場合は、全依存物と素材の利用条件を確認し、ライセンスを明示する。

## D-002: 審査員向け資料は英語、内部運用は日本語にする

- 日付: 2026-07-20
- 状態: 採用
- 判断: README、提出説明、動画台本、審査証拠表は英語で作成し、内部の手順と判断記録は日本語で作成する。
- 理由: 提出要件を満たしつつ、日々の作業で誤読を減らすため。

## D-003: 動画は公開設定にする

- 日付: 2026-07-20
- 状態: 採用
- 判断: YouTube 動画は限定公開ではなく公開とする。
- 理由: 告知と Official Rules の記述差を安全側で解消するため。

## D-004: 文書を主役、質問を補助にする

- 日付: 2026-07-21
- 状態: 採用
- 判断: Product Constitution を中央の主要成果物とし、AI との会話や質問は、文書を確定するための補助領域に置く。
- 選択肢: 会話履歴を中心にする、設定フォームを中心にする、文書を中心にする。
- 理由: 非技術者でも、何が決まり、誰が承認し、何が未決かを会話履歴を読み返さず把握できるため。
- 不利益: 文書構造と出所表示の設計が必要になり、単純な会話画面より実装量が増える。
- 再確認条件: 利用試験で、文書より会話の方が意図の確認速度と正確さで明確に優れる場合。

## D-005: shadcn/ui の Base UI 版を部品基盤にする

- 日付: 2026-07-21
- 状態: 採用
- 判断: `shadcn` 4.13.1、`base-nova`、Base UI を使い、製品固有の画面は公式部品の組合せとして実装する。
- 選択肢: 全部品を独自実装する、別の完成済み部品群を使う、shadcn/ui を使う。
- 理由: 読み上げ、キーボード操作、重ね表示、入力、状態表示の既知の作法を使いながら、CriteriaForge 固有の視覚言語を意味変数で表現できるため。
- 不利益: 旧版の Radix UI 向け例と現行 Base UI の構成を混ぜない注意が必要。
- 再確認条件: 公式部品では中核操作の読み上げ要件や表示性能を満たせない場合。

## D-006: 「文書工房 × 精密検査器」を視覚方向にする

- 日付: 2026-07-21
- 状態: 採用
- 判断: 温かい生成り色の紙面、墨色、明朝系の文書見出しを基調とし、承認を熾火色、根拠を青、人間確認済みを緑で示す。
- 選択肢: 一般的な業務用青色画面、会話アプリ風、文書工房と検査器の組合せ。
- 理由: 「意図を憲法へ編む」と「成果物を同じ基準で検査する」という二つの仕事を直感的に結び付けられるため。
- 不利益: 装飾を増やすと可読性を損なうため、影、質感、動きを抑制する必要がある。
- 再確認条件: 低視力、色覚、長文読解の試験で可読性上の問題が見つかった場合。

## D-007: 狭い画面では補助領域を画面外パネルへ移す

- 日付: 2026-07-21
- 状態: 採用
- 判断: デスクトップは文書と質問を同時表示し、スマートフォンでは文書を残して質問を `Sheet`、作業循環を画面外 `Sidebar` にする。
- 選択肢: 三列を縮小する、スマートフォンを対象外にする、補助領域を移す。
- 理由: 文書の可読幅と押しやすい操作対象を維持しながら、同じ情報構造を保てるため。
- 不利益: デスクトップより一操作増える。
- 再確認条件: スマートフォンで質問と文書の往復が主要な離脱要因になる場合。

## D-008: ChatGPT OAuthをCodex CLI経由で再利用する

- 日付: 2026-07-21
- 状態: 採用
- 判断: `codex login status`と`codex exec`を使い、認証ファイルやトークン本文は読まない。
- 理由: 利用者にAPIキーを要求せず、独自OAuthクライアントも持たないため。
- 不利益: Codex CLIの導入とログインが必要で、版による挙動差を実行記録へ固定する必要がある。

## D-009: 公開体験版は実測結果の再生にする

- 日付: 2026-07-21
- 状態: 採用
- 判断: 架空データだけを用い、実際の`gpt-5.6-sol`3回評価を記録して再生する。ライブ実行とは表示しない。
- 理由: 無認証公開と私的データ非保存を両立し、審査員が結果と再現情報を確認できるため。
- 不利益: 任意入力に対するライブ評価は公開版では試せない。

## D-010: AIの揺れを多数決で隠さない

- 日付: 2026-07-21
- 状態: 採用
- 判断: 必須判定、適用範囲、根拠充足、判断不能が一致しない場合は`blocked`として人間へ返す。
- 理由: 2対1の多数決では、基準の曖昧さや根拠不足を見えなくするため。
- 不利益: 自動的に結論へ到達しない案件が増える。

## D-011: 公開版の構築時にローカルAPIを除く

- 日付: 2026-07-21
- 状態: 採用
- 判断: 配備用構築では`src/app/api`を構築対象から外し、成果物を検査する。
- 理由: 実行時の403だけに頼らず、SQLite、ファイル、子プロセス、OAuth経路を公開成果物へ入れないため。
- 不利益: 通常のローカル本番構築と公開配備用構築を別々に検査する必要がある。

## D-012: 提出動画をRemotionのフレーム駆動構成にする

- 日付: 2026-07-21
- 状態: 採用
- 判断: 提出動画は、実画面、時間合わせ済みの英語音声、字幕をRemotionで統合し、すべての動きを`useCurrentFrame()`に基づいて再現可能に書き出す。日本語訳は本人確認用として`output/`配下だけに保存する。確認用書き出しのmacOS合成音声は公開利用条件を確定できないため、一般公開前に本人録音または条件を確認できる音声へ差し替える。
- 選択肢: 静止画をFFmpegで連結する、画面録画を手作業で編集する、Remotionで意味と時間をコード化する。
- 理由: 3回評価、権限分離、不可逆な憲法、一時Git作業領域、不一致時の停止という抽象的な関係を、製品の視覚体系を保った動きとして示せるため。音声、字幕、画面の同期もフレーム番号で再検証できる。
- 不利益: 書き出し時間と依存物が増え、映像用のソースを製品本体とは別に保守する必要がある。
- 再確認条件: 第三者端末で字幕が読めない、動きが実画面の理解を妨げる、180秒未満の条件を満たせない、提出主体が個人ではなくなる、Remotionの版を変更する、または音声の公開条件を確認できない場合。
